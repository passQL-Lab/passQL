# ⚙️[기능추가][Quiz] 실시간 AI 선택지 세트 생성 및 프리페치 아키텍처

**라벨**: `작업전`
**담당자**:

---

📝현재 문제점
---

- 현재 문제(question) 구조는 선택지(choice)가 문제에 고정되어 있어, 동일 문제를 여러 번 풀 때 정답 패턴이 외워지는 문제가 있다.
- AI 기반 SQL 학습 서비스의 차별점(매번 다른 선택지, 실시간 생성)을 살리지 못하고 있다.
- `question` 테이블이 `answer_sql` 중심으로 재설계되지 않아, 샌드박스 검증 기반 정답 판정이 불가능하다.
- 선택지 세트의 상태·출처·재사용 여부를 관리할 데이터 모델이 없어, 향후 CURATED/HYBRID/신고 처리로 확장할 경로가 막혀 있다.
- 문제 진입 시마다 AI를 호출하면 체감 지연이 커지는데, 이를 완화할 프리페치 전략이 정의되어 있지 않다.

🛠️해결 방안 / 제안 기능
---

설계 문서 `docs/superpowers/specs/2026-04-09-realtime-choice-set-generation-design.md` 기반으로 다음을 구현한다.

**1. 핵심 아키텍처 — 실시간 생성 + 프리페치**
- 문제(question)는 고정, 선택지 세트(question_choice_set)는 퀴즈 진입 시마다 AI로 생성.
- 현재 N번 문제를 푸는 동안 N+1번 문제의 선택지 세트를 백그라운드에서 미리 생성하여 체감 지연 최소화.
- MVP 정책: `AI_ONLY` 단일 모드. `CURATED_ONLY` / `HYBRID`는 enum·컬럼만 예약.

**2. 데이터 모델 (Flyway V0_0_28)**
- `question`: `answer_sql` 중심으로 재설계, `choice_set_policy` 컬럼 추가.
- `question_choice_set`: `source`, `status`, `member_id`, `raw_response_json`, `consumed_at`, `is_reusable` 포함.
- `question_choice_set_item`: 개별 선택지 항목.
- `quiz_session`, `submission` 확장: 세션 단위 선택지 세트 추적.

**3. AI 호출 경로**
- 메인: Spring → Python AI Server (Gemini 2.5 flash-lite).
- Fallback: Spring 내장 `GeminiClient` 직접 호출.
- Ollama는 embedding 전용으로 한정.

**4. 샌드박스 검증 기반 정답 판정**
- `answer_sql` 실행 결과가 진실(ground truth).
- 선택지 세트의 `correctCount != 1`이면 최대 3회 재생성.
- 3회 모두 실패 시 `status = FAILED`로 저장하고 사용자에겐 재시도 안내.

**5. 관리자 기능**
- AI 보조 문제 등록: 풀 스키마 + INSERT를 AI에 전달하여 `answer_sql` 초안 생성.
- 선택지 세트 대시보드: 상태/재사용/비활성 관리.
- 프롬프트 테스트 페이지 연계 (`2026-04-08-admin-question-generation-prompt-test-design.md` 참조).

**6. 확장성 확보**
- `source` 다값 enum, `is_reusable` 플래그, `status = REPORTED` 예약.
- MVP에서는 실제 동작은 `AI_ONLY`만 수행하되 스키마·코드 분기점을 미리 마련.

**7. 에러 처리 통일**
- 모든 에러는 `CustomException(ErrorCode.XXX)` 단일 패턴으로 통일.
- 신규 ErrorCode 추가 항목은 설계 문서의 에러 섹션 참조.
