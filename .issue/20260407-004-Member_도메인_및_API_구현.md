⚙️ [기능추가][Server] Member 도메인 모듈 신규 생성 및 확장성 있는 Entity 설계

📝 현재 문제점
---

- 현재 PRD 데이터 모델에는 `submission`, `execution_log` 등 활동 테이블만 존재하고, 사용자 식별 주체인 `member` 도메인이 정의되어 있지 않습니다.
- 프론트엔드는 로그인 없이 동작하지만, 사용자별 진도/제출 이력을 추적하려면 백엔드에서 발급한 `member_uuid`와 닉네임이 필요합니다.
- 닉네임 단어 풀을 직접 DB로 관리하면 시드 데이터 유지보수 부담이 크고, 조합 로직도 별도로 구현해야 합니다. 이미 자체 라이브러리 `suh-random-engine`(`me.suhsaechan:suh-random-engine`)이 형용사+명사 약 96만 조합을 제공하므로, 이를 활용해 일원화하는 것이 합리적입니다.
- 현재 멀티모듈 구조(`PQL-Domain-Meta`, `PQL-Domain-Question`, `PQL-Domain-Submission`, `PQL-Domain-AI`)에 Member 도메인이 빠져 있어, 추후 다른 도메인이 회원 정보를 참조할 때 의존성이 꼬일 수 있습니다.
- 익명 사용자 전제이지만 향후 SNS 로그인/계정 연동/관리자 권한 같은 확장 가능성을 열어두지 않으면 단기간 내 마이그레이션 부담이 커집니다.

🛠️ 해결 방안 / 제안 기능
---

- 멀티모듈 구조를 따라 `PQL-Domain-Member` 모듈을 신규로 생성하고, Member 도메인을 독립 모듈로 분리합니다.
- `Member` Entity는 익명 사용자 기본 동작을 보장하면서도, 향후 인증 수단/권한/상태/탈퇴/감사 정보 확장이 자연스럽도록 컬럼을 미리 구조화합니다.
- 닉네임 생성은 자체 라이브러리 `suh-random-engine`(`me.suhsaechan:suh-random-engine:1.1.0`)의 `SuhRandomKit`을 사용하여 일원화하며, 중복 회피·재생성 로직만 `MemberService`로 캡슐화합니다. (자체 단어 테이블/시드/어드민 CRUD 불필요)
- D2 단계에서는 익명 등록만 지원하지만, Entity/스키마는 OAuth 연동·다중 계정·역할 관리까지 수용 가능한 형태로 미리 설계합니다.

🧱 Member Entity 확장 설계 (제안)
---

- 식별/기본
  - `member_uuid` CHAR(36) PK — 외부 노출 식별자
  - `nickname` VARCHAR(50) UNIQUE NOT NULL
- 인증/연동 (확장 여지)
  - `auth_provider` VARCHAR(30) — `ANONYMOUS` / `GOOGLE` / `KAKAO` / ... (기본 `ANONYMOUS`)
  - `provider_user_id` VARCHAR(255) NULLABLE — provider별 외부 식별자
  - `email` VARCHAR(255) NULLABLE
  - `email_verified` TINYINT(1) DEFAULT 0
- 권한/역할
  - `role` VARCHAR(30) — `USER` / `ADMIN` (기본 `USER`)
- 상태/라이프사이클
  - `status` VARCHAR(20) — `ACTIVE` / `DORMANT` / `WITHDRAWN` (기본 `ACTIVE`)
  - `withdrawn_at` DATETIME(6) NULLABLE
- 행동 추적
  - `last_seen_at` DATETIME(6)
  - `last_login_ip` VARCHAR(45) NULLABLE — IPv6 대응
- 감사 정보 (BaseEntity 상속으로 자동 처리)
  - `created_at`, `updated_at`, `created_by`, `updated_by`
- 인덱스
  - UNIQUE(`nickname`)
  - INDEX(`auth_provider`, `provider_user_id`) — 향후 OAuth 조회용
  - INDEX(`status`)

- [ ] `MemberService`
