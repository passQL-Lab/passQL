📝현재 문제점
---

### 1. 스키마 표시가 읽기 어려움

현재 "스키마 보기" 아코디언을 열면 아래 3가지가 텍스트 덤프로 나옴:
- **테이블 구조**: `CUSTOMER (id INT PK, name VARCHAR, email VARCHAR, ...)` — 한 줄로 나열
- **DDL**: `CREATE TABLE CUSTOMER (id INT PRIMARY KEY, ...)` — 코드 그대로
- **샘플 데이터**: `CUSTOMER: (1, '홍길동', 'hong@test.com', ...)` — 튜플 텍스트

**문제**: 모바일에서 가로 스크롤이 필요하고, 실제 테이블 형태로 보이지 않아 문제를 이해하기 어려움. SQLD 시험에서는 테이블 다이어그램이 제공되는데 현재 passQL은 원시 텍스트만 보여줌.

### 2. 선택지 클릭 후 실행 결과 표시 부족

현재 EXECUTABLE 모드에서 선택지 클릭 시:
- 성공: 행 수 + 실행 시간만 간략히 표시, 결과 테이블은 작게 나옴
- 에러: 에러 코드 + 메시지만 텍스트로 표시

**문제**: 
- 결과 테이블이 선택지 카드 안에 갇혀 있어 데이터를 비교하기 어려움
- 여러 선택지의 실행 결과를 나란히 비교할 수 없음
- "이 SQL이 왜 맞는지/틀렸는지"를 결과 데이터로 직접 느끼는 체험이 부족

### 3. 제출 후 결과 비교 화면 없음

현재 `AnswerFeedback` 페이지:
- 정답: 초록 배경 + "정답입니다!" + 정답 SQL + 해설 텍스트
- 오답: 빨강 배경 + 내 SQL + 정답 SQL + 해설 텍스트

**문제**: 
- **실행 결과 테이블 비교가 없음** — "내 SQL은 이런 결과를 뱉고, 정답 SQL은 이런 결과를 뱉는구나" 비교 불가
- CONCEPT_ONLY 문제에서도 정답/오답 SQL이 표시됨 (SQL이 아닌데)
- passQL의 핵심 차별점인 "눈으로 보고 비교하는 재미"가 실현되지 않음

### 4. 자유 SQL 실행 기능 없음

- 문제 스키마 위에서 사용자가 자유롭게 SQL을 입력/실행해볼 수 있는 인터페이스가 없음
- BE에 `POST /api/questions/{uuid}/execute` (body: `{sql}`)가 이미 구현되어 있으나, FE에서 선택지 자동 실행 용도로만 사용 중
- 사용자가 "이 테이블에서 내가 직접 쿼리 짜보고 싶다"는 욕구를 충족할 수 없음

### 1. 스키마 시각화 개선

**현재 텍스트 덤프 대신 실제 테이블 형태로 표시**

#### 1-1. 테이블 구조 카드
```
┌─ CUSTOMER ──────────────────────────┐
│ id       │ INT     │ PK            │
│ name     │ VARCHAR │               │
│ email    │ VARCHAR │               │
└─────────────────────────────────────┘

┌─ ORDERS ────────────────────────────┐
│ id          │ INT  │ PK            │
│ customer_id │ INT  │ FK→CUSTOMER   │
│ amount      │ INT  │               │
│ order_date  │ DATE │               │
└─────────────────────────────────────┘
```

- 각 테이블을 카드 형태로 표시 (테이블명 헤더 + 컬럼 목록)
- PK는 열쇠 아이콘, FK는 링크 아이콘으로 시각적 구분
- `schemaDisplay` 필드를 파싱하거나, BE에서 구조화된 JSON으로 제공

#### 1-2. 샘플 데이터 테이블
```
CUSTOMER
| id | name   | email           |
|----|--------|-----------------|
| 1  | 홍길동 | hong@test.com   |
| 2  | 김영희 | kim@test.com    |

ORDERS  
| id | customer_id | amount | order_date |
|----|-------------|--------|------------|
| 1  | 1           | 50000  | 2026-01-01 |
| 2  | 1           | 30000  | 2026-01-15 |
```

- `schemaSampleData`의 INSERT문을 파싱하여 실제 테이블 렌더링
- 또는 BE에서 `POST /api/questions/{uuid}/execute`로 `SELECT * FROM {table} LIMIT 5`를 실행하여 결과 반환

#### 1-3. DDL 접기
- DDL 원문은 "DDL 보기" 토글 뒤에 숨김 (기본 접힘)
- 주로 학습 목적으로 DDL을 보고 싶은 사용자만 열어봄

### 2. 선택지 실행 결과 개선

#### 2-1. 선택지 카드 내 결과 테이블 확장

현재 선택지 카드 안에 결과가 갇혀 있는데, 터치 시 결과를 **풀 와이드 영역**으로 확장:

```
┌─ A (선택됨) ──────────────────────┐
│ SELECT c.name, COUNT(*) AS cnt   │
│ FROM CUSTOMER c                   │
│ JOIN ORDERS o ON c.id = o.cust... │
├───────────────────────────────────┤
│ 실행 결과 (3행, 34ms)              │
│ ┌──────┬─────┐                    │
│ │ name │ cnt │                    │
│ ├──────┼─────┤                    │
│ │ 홍길동│  2  │                    │
│ │ 김영희│  3  │                    │
│ │ 이철수│  1  │                    │
│ └──────┴─────┘                    │
└───────────────────────────────────┘
```

#### 2-2. 에러 시 시각적 구분 강화

```
┌─ B ──────────────────────────────┐
│ SELECT c.name, COUNT(*) AS cnt   │
│ FROM CUSTOMER c                   │
│ JOIN ORDERS o ON c.id = o.cust_id│  ← 에러 지점 하이라이트
├───────────────────── ERROR ──────┤
│ Unknown column 'o.cust_id'       │
│                                   │
│  [AI에게 물어보기]                │
└───────────────────────────────────┘
```

### 3. 제출 후 결과 비교 화면 (AnswerFeedback 재설계)

#### 3-1. EXECUTABLE 문제 결과 화면

**오답 시 — 핵심 경험: "내 결과 vs 정답 결과 나란히 비교"**

```
  오답이에요
  정답: A

  ── 내가 선택한 SQL (B) ──────────────
  SELECT c.name, COUNT(*) AS cnt
  FROM CUSTOMER c
  JOIN ORDERS o ON c.id = o.cust_id  
  GROUP BY c.name
  
  결과: ERROR
  Unknown column 'o.cust_id' in 'on clause'

  ── 정답 SQL (A) ────────────────────
  SELECT c.name, COUNT(*) AS cnt
  FROM CUSTOMER c  
  JOIN ORDERS o ON c.id = o.customer_id
  GROUP BY c.name

  결과:
  | name   | cnt |
  |--------|-----|
  | 홍길동 |  2  |
  | 김영희 |  3  |
  | 이철수 |  1  |

  ── 해설 ─────────────────────────────
  ORDERS 테이블의 외래키 컬럼명은 customer_id입니다...

  [AI에게 자세히 물어보기]
  [다음 문제]
```

**BE 의존**: `SubmitResult`에 `selectedResult`, `correctResult` (ExecuteResult) 추가 필요 (#008 이슈 참조)

#### 3-2. CONCEPT_ONLY 문제 결과 화면

```
  오답이에요
  정답: C

  ── 내가 선택한 답 (A) ──────────────
  INNER JOIN은 양쪽 테이블에 매칭되는 행만 반환한다

  ── 정답 (C) ────────────────────────
  LEFT JOIN은 왼쪽 테이블의 모든 행을 반환한다
  
  ── 해설 ─────────────────────────────
  LEFT JOIN은 왼쪽 테이블 기준으로...

  [다음 문제]
```

실행 결과 테이블 없이 텍스트 비교만 표시.

### 4. 자유 SQL 실행 (SQL Playground)

문제 화면 하단에 **접이식 SQL 입력 영역** 추가:

```
  ── SQL 실행기 ──── [접기 ^] ──────────
  ┌─────────────────────────────────────┐
  │ SELECT * FROM CUSTOMER              │
  │ WHERE id = 1;                       │
  │                                     │
  └─────────────────────────────────────┘
  [실행]                    0.034s

  | id | name   | email           |
  |----|--------|-----------------|
  | 1  | 홍길동 | hong@test.com   |
```

**동작**:
- 기존 BE API `POST /api/questions/{uuid}/execute` 그대로 사용 (body: `{sql}`)
- 모바일: 바텀시트로 열림 (키보드 친화적)
- 데스크톱: 하단 패널
- 실행 결과는 같은 테이블 컴포넌트로 렌더링
- SELECT만 허용 (SandboxExecutor가 이미 안전하게 처리)
- "이 스키마에서 이렇게 쿼리하면 어떤 결과가 나올까?" 직접 확인 가능
