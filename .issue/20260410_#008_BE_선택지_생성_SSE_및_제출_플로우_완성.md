# ⚙️[기능추가][BE] 사용자 문제 풀기 선택지 생성(SSE) 및 제출 플로우 완성

**라벨**: `작업전`
**담당자**: 

---

📝현재 문제점
---

### 1. 선택지 생성 SSE 엔드포인트 미구현

- FE에서 `POST /api/questions/{questionUuid}/generate-choices` (SSE) 호출 코드가 완성되어 있으나 **BE에 해당 엔드포인트가 없음**
- 관리자가 등록한 ADMIN_SEED 선택지는 `GET /api/questions/{uuid}`의 `choiceSets[]`에 포함되어 반환되지만, **실시간 AI 선택지 생성 경로가 없음**
- prompt_template에 `generate_choice_set` 템플릿은 시드 완료 (V0_0_28), AiGatewayClient도 구현되어 있으나 **연결하는 Controller/Service가 없음**

### 2. SubmissionService가 구 QuestionChoice 참조

- `SubmissionService.submit()`이 `QuestionChoiceRepository.findByQuestionUuidAndChoiceKey()`로 **구 question_choice 테이블**을 참조함
- 신규 `QuestionChoiceSetItem`을 전혀 사용하지 않음
- `Submission` 저장 시 `choiceSetUuid`가 항상 NULL

### 3. SubmitRequest에 choiceSetId 없음

- FE는 `{ choiceSetId, selectedChoiceKey }` 를 전송하려 구현되어 있음
- BE `SubmitRequest`는 `selectedChoiceKey`만 받음 — **FE/BE 불일치**

### 4. 실행 결과 비교 응답 부재

- 현재 `SubmitResult`는 `{ isCorrect, correctKey, rationale }` 3개 필드만 반환
- 사용자가 선택한 SQL의 실행 결과와 정답 SQL의 실행 결과를 **비교해서 보여줄 데이터가 없음**
- passQL의 핵심 차별점인 "눈으로 보고 비교하는 재미"가 구현 불가

🛠️해결 방안 / 제안 기능
---

### 1. SSE 선택지 생성 엔드포인트 구현

**엔드포인트**: `POST /api/questions/{questionUuid}/generate-choices`

**요청**:
```
Headers:
  X-Member-UUID: {memberUuid}
  Accept: text/event-stream
  Content-Type: application/json
```

**SSE 이벤트 프로토콜** (FE가 이미 기대하는 형식):
```
event: status
data: {"phase":"generating","message":"선택지 생성 중..."}

event: status  
data: {"phase":"validating","message":"SQL 실행 검증 중..."}

event: complete
data: {"choiceSetId":"uuid-string","choices":[{"key":"A","kind":"SQL","body":"SELECT ...","sortOrder":1},{"key":"B",...},{"key":"C",...},{"key":"D",...}]}

-- 또는 오류 시:
event: error
data: {"code":"GENERATION_FAILED","message":"선택지 생성에 실패했습니다","retryable":true}
```

**서버 동작 흐름**:
```
1. question 조회 → choiceSetPolicy 확인
2. policy별 분기:
   - AI_ONLY: AiGatewayClient로 선택지 생성 요청
   - CURATED_ONLY: 기존 ADMIN_SEED/ADMIN_CURATED 세트 중 하나 반환 (MVP에선 미사용)
3. AI 응답 수신 → QuestionChoiceSet (source=AI_RUNTIME) + Items 저장
4. EXECUTABLE이면: 4개 선택지 SQL을 SandboxExecutor로 실행 → 정답 자동 판별
5. 정답이 정확히 1개인지 검증 (아니면 최대 3회 재생성)
6. SSE로 complete 이벤트 전송
```

**반환하는 ChoiceItem에 isCorrect/rationale 포함하지 않음** (사용자에게 정답 노출 방지)

### 2. SubmitRequest 확장 + SubmissionService 마이그레이션

**SubmitRequest 변경**:
```java
public record SubmitRequest(
    UUID choiceSetId,           // 신규: 어떤 선택지 세트에서 골랐는지
    String selectedChoiceKey    // 기존: A/B/C/D
) {}
```

**SubmissionService.submit() 변경**:
```
1. choiceSetId로 QuestionChoiceSet 조회
2. QuestionChoiceSetItem에서 selectedChoiceKey 찾기
3. isCorrect 판별
4. Submission 저장 (choiceSetUuid 포함)
5. 구 QuestionChoice 참조 제거
```

### 3. SubmitResult 확장 (실행 결과 비교)

**SubmitResult 변경**:
```java
public record SubmitResult(
    boolean isCorrect,
    String correctKey,
    String rationale,
    // 신규 필드들:
    ExecuteResult selectedResult,   // 사용자가 고른 SQL 실행 결과
    ExecuteResult correctResult,    // 정답 SQL 실행 결과
    String correctSql,              // 정답 SQL 본문
    String selectedSql              // 선택한 SQL 본문
) {}
```

**동작**: EXECUTABLE 문제일 때 submit 시점에 양쪽 SQL을 SandboxExecutor로 실행하여 결과를 함께 반환. CONCEPT_ONLY면 `selectedResult`, `correctResult`는 null.

### 4. 자유 SQL 실행 지원

현재 `POST /api/questions/{questionUuid}/execute`는 body의 `sql` 필드로 **임의 SQL을 실행**할 수 있는 구조가 이미 되어 있음. 단, 현재는 선택지 SQL만 실행하는 용도로 사용 중.

**추가 필요 사항**:
- 사용자가 자유 SQL을 입력하여 실행하는 것도 이 엔드포인트로 처리 가능
- BE 변경 불필요 (이미 범용적으로 구현됨)
- FE에서 자유 입력 UI + 이 엔드포인트 호출만 하면 됨

  - `QuestionChoiceSet` (source=AI_RUNTIME) + `QuestionChoiceSetItem` 저장

  - `event: complete` (choiceSetId + choices)
  - `event: error` (retryable 여부)
- [ ] ChoiceItem 응답에서 `isCorrect`, `rationale` 제외 (사용자에게 정답 노출 방지)
- [ ] 프리페치 지원: memberUuid로 선택지 세트 사전 생성, `generatedForMemberUuid` 필드 활용

### Phase 2: 제출 플로우 마이그레이션

- [ ] `SubmitRequest`에 `choiceSetId` (UUID) 필드 추가
- [ ] `SubmissionService.submit()` 리팩토링
  - `QuestionChoiceSetItem`에서 정답 조회 (구 `QuestionChoice` 참조 제거)
  - `Submission.choiceSetUuid` 실제 값 저장
- [ ] `SubmitResult` 확장
  - `selectedResult`, `correctResult` (ExecuteResult) 추가
  - `correctSql`, `selectedSql` (String) 추가
  - EXECUTABLE 문제: submit 시 양쪽 SQL 실행하여 결과 포함
  - CONCEPT_ONLY 문제: 실행 결과 필드 null

### Phase 3: 에러 처리

- [ ] `ErrorCode` 추가
  - `CHOICE_SET_NOT_FOUND`: 선택지 세트 미존재
  - `CHOICE_SET_GENERATION_FAILED`: AI 생성 실패 (3회 재시도 후)
  - `CHOICE_SET_ITEM_NOT_FOUND`: 선택지 키 미존재
  - `CHOICE_SET_ALREADY_CONSUMED`: 이미 제출된 세트
- [ ] SSE 에러 시 `retryable` 플래그로 FE 재시도 유도
