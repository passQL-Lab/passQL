# ❗[버그][문제풀기] 선택지별 쿼리 실행 버튼이 독립적으로 동작하지 않는 버그

## 개요

문제 풀기 화면(제출 후 실행 모드)에서 선택지 A가 "실행 중..." 상태일 때 선택지 B의 "쿼리 실행"을 누르면 A의 상태가 초기화되는 버그를 수정했다. `useMutation` 단일 인스턴스의 상태(`isPending`, `variables`)에 의존하던 방식을 `executingKeys` Set 기반 독립 추적으로 전환하고, `mutateAsync`로 병렬 호출 시 콜백 누락 문제까지 함께 해결했다.

## 변경 사항

### 프론트엔드
- `client/src/pages/QuestionDetail.tsx`: `executingKeys` Set state 및 ref 추가, `handleExecute` async 전환, `isExecuting` props 계산 로직 교체, `questionUuid` 변경 시 초기화 추가

## 주요 구현 내용

**기존 방식 (버그 원인)**

`isExecuting` props를 `executeMutation.isPending && executeMutation.variables === choice.body`로 계산했다. `useMutation` 인스턴스는 마지막 호출의 상태 하나만 유지하므로, A 실행 중 B를 호출하면 `variables`가 B의 SQL로 덮어씌워져 A 카드가 "쿼리 실행"으로 리셋됐다. 또한 `mutate()`에 콜백을 전달하는 방식은 병렬 호출 시 마지막 콜백만 보장되어 앞선 호출의 `onSuccess`가 무시될 수 있었다.

**개선 방식**

- `executingKeys: Set<string>` state로 현재 실행 중인 choiceKey 목록을 독립 관리
- `executingKeysRef`로 클로저 내 stale state 없이 최신 값 참조
- `mutate()` → `mutateAsync()` + try/finally 전환으로 각 호출이 독립된 Promise를 반환하게 해 병렬 호출 시 콜백 누락 방지
- `finally`에서 항상 해당 key를 Set에서 제거해 성공/에러 모두 상태 정리 보장
- `questionUuid` 변경 시 `executingKeys` 초기화 추가 (practiceMode에서 리마운트 없이 문제가 교체되는 케이스 대응)

## 주의사항

- HTTP 에러(네트워크 오류, 서버 500 등) 발생 시 `catch` 블록은 묵음 처리한다. SQL 실행 실패는 HTTP 200 + `errorCode` 포함 응답으로 내려오므로 `ResultTable`이 에러 카드를 표시하는 정상 경로로 처리된다. HTTP 에러는 `finally`에서 버튼이 "쿼리 실행"으로 복구되어 재시도 가능하다.
- `AnswerFeedback.tsx`도 동일한 Set 패턴을 사용 중이며, 이번 수정으로 두 화면의 실행 상태 관리 방식이 일관성을 갖게 됐다.
