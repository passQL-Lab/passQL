# [기능추가][통계] 토픽별 분석 및 AI 코멘트 API 구현

## 개요

"내 실력, 한눈에" 통계 화면을 위한 2개의 신규 API를 추가했다. 기존 `/api/progress`가 제공하지 않던 **토픽별 정답률(레이더 차트)**, **토픽별 문제 수(가로 막대 그래프)**, **AI 영역 분석 코멘트**를 `GET /api/progress/topic-analysis`, `GET /api/progress/ai-comment` 엔드포인트로 분리 제공한다. AI 코멘트는 Redis에 24시간 캐싱하며, 새 풀이 제출 시 자동으로 무효화된다.

## 변경 사항

### DTO 추가
- `PQL-Domain-Submission/.../dto/TopicStat.java`: 토픽 UUID, 표시명, 전체 문제 수, 정답률, 풀이 수를 담는 record
- `PQL-Domain-Submission/.../dto/TopicAnalysisResponse.java`: `List<TopicStat>` 래핑 record
- `PQL-Domain-AI/.../dto/AiCommentResponse.java`: AI 코멘트 문자열 + 생성 시각 record

### Repository 쿼리 추가
- `PQL-Domain-Submission/.../repository/SubmissionRepository.java`
  - `findTopicStatsAfter()`: 최근 7일 내 제출 기준, 문제별 최근 1회만 추려(`ROW_NUMBER() PARTITION BY question_uuid`) 토픽별 풀이 수·정답률 집계하는 네이티브 쿼리
- `PQL-Domain-Question/.../repository/QuestionRepository.java`
  - `countActiveByTopic()`: 활성 문제(`is_active = true`) 토픽별 총 개수 집계 쿼리

### 서비스 추가
- `PQL-Domain-Submission/.../service/TopicAnalysisService.java`
  - SubmissionRepository·QuestionRepository 결과를 TopicRepository 전체 활성 토픽 기준으로 머지 → 풀이 이력이 없는 토픽도 0으로 포함
- `PQL-Application/.../service/AiCommentService.java`
  - Redis 캐시 조회 → miss 시 GeminiClient 호출(system/user 프롬프트 분리) → 응답 Redis 저장(TTL 24h)
  - `evictCache(memberUuid)`: 캐시 강제 삭제 메서드 (Submit 완료 시 호출용)

### 기존 서비스 수정

  - `submit()` 완료 후 `aiCommentService.evictCache(memberUuid)` 호출 → 새 제출이 생기면 캐시 무효화

### 컨트롤러·문서 수정

- `PQL-Web/.../controller/ProgressControllerDocs.java`: Swagger 문서 업데이트

### 테스트

## 주요 구현 내용

### 토픽별 정답률 계산 방식
단순 전체 제출 평균이 아닌 **문제별 최근 시도 1회**만 사용한다. `ROW_NUMBER() OVER (PARTITION BY question_uuid ORDER BY submitted_at DESC)`로 최신 제출만 추린 뒤 토픽별 정답률을 계산해 반복 연습으로 인한 정답률 왜곡을 방지한다.

### AI 코멘트 캐싱 전략
Submission count 비교 대신 **이벤트 기반 evict** 방식을 채택했다. `SubmissionService.submit()` 완료 시 `evictCache()`를 직접 호출하므로 별도 카운트 컬럼 없이 단순하게 무효화할 수 있다. TTL 24시간을 추가로 두어 evict가 누락되더라도 하루 뒤 자동 갱신된다.

### FE 연동 방식 (E2E 흐름)

통계 화면 진입 시 아래 순서로 3개 API를 호출한다.

```
1. GET /api/progress?memberUuid={uuid}
   → ProgressResponse { solvedCount, correctRate, streakDays, readiness }
   → 상단 요약 카드 (310문제 / 42% / 3일)

2. GET /api/progress/topic-analysis?memberUuid={uuid}
   → TopicAnalysisResponse {
       topicStats: [
         { topicUuid, displayName, totalQuestionCount, correctRate, solvedCount },
         ...
       ]
     }
   → 레이더 차트: displayName + correctRate 사용
   → 가로 막대 그래프: displayName + totalQuestionCount 사용

3. GET /api/progress/ai-comment?memberUuid={uuid}         ← 비동기 로딩 권장
   → AiCommentResponse { comment, generatedAt }
   → "영역 분석" 카드의 코멘트 텍스트
```

**비동기 로딩 권장**: `/ai-comment`는 캐시 miss 시 Gemini 외부 호출로 응답이 느릴 수 있다. 1·2번 API를 먼저 렌더링하고 3번은 별도 비동기 요청으로 나중에 채우는 방식을 권장한다.

**캐시 갱신 조건**: 문제를 풀고 제출하면 서버에서 자동으로 캐시를 삭제한다. 다음 `/ai-comment` 호출 시 최신 데이터 기준으로 새 코멘트가 생성된다.

## 응답 예시

### `GET /api/progress/topic-analysis?memberUuid=...`
```json
{
  "topicStats": [
    {
      "topicUuid": "xxxxxxxx-...",
      "displayName": "JOIN",
      "totalQuestionCount": 45,
      "correctRate": 0.72,
      "solvedCount": 11
    },
    {
      "topicUuid": "yyyyyyyy-...",
      "displayName": "서브쿼리",
      "totalQuestionCount": 30,
      "correctRate": 0.40,
      "solvedCount": 5
    }
  ]
}
```

### `GET /api/progress/ai-comment?memberUuid=...`
```json
{
  "comment": "전반적으로 안정적인 실력을 보여주고 있어요. 실력을 더 높이고 싶다면 윈도우 함수, ORDER BY, GROUP BY 카테고리를 집중적으로 학습해보세요.",
  "generatedAt": "2026-04-10T14:32:00"
}
```

## 주의사항

- `correctRate`는 0~1 범위(소수)로 반환된다. FE에서 `%` 표시 시 `× 100` 변환 필요.
- `totalQuestionCount`는 **전체 활성 문제 수**이며, `solvedCount`는 **최근 7일 내 풀이 수**라 `totalQuestionCount > solvedCount`가 정상이다.
- `/ai-comment` 첫 호출(캐시 cold start)은 Gemini 응답 시간만큼 지연될 수 있다. 로딩 스피너 처리 필요.
- 풀이 이력이 전혀 없는 신규 회원의 경우 `correctRate: 0.0`, `solvedCount: 0`으로 모든 토픽이 반환된다.
