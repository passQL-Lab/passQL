# ⚙️[기능추가][Auth] OAuth + JWT 인증 시스템 구현

## 개요

Firebase Admin SDK 기반 OAuth 소셜 로그인(5개 provider)과 Spring Security JwtAuthenticationFilter를 passQL 백엔드에 전면 적용했다. 기존 익명 회원(`ANONYMOUS`) 방식을 완전히 제거하고, AccessToken(1h) + RefreshToken(14d, Redis) 이중 토큰 구조로 전환했다. 모든 보호 API의 `memberUuid` 쿼리 파라미터를 `@AuthMember` 토큰 기반으로 전환했다.

## 주요 구현 내용

**인증 플로우**: 프론트가 Firebase SDK로 소셜 로그인 후 idToken을 `POST /api/auth/login`으로 전달 → `SocialLoginStrategy`로 provider별 검증 → 신규면 `Member.signUp()` 가입 처리 → AccessToken + RefreshToken 발급, RefreshToken은 Redis에 저장.

**Strategy Pattern**: 5개 provider 중 Google/Apple은 Firebase Admin SDK `verifyIdToken()`을 공통 추상 클래스(`FirebaseLoginStrategy`)로 처리하고, Kakao/Naver/Github은 각 provider REST API를 직접 호출하는 독립 구현체로 분리했다. `SocialLoginStrategyConfig`가 `Map<AuthProvider, SocialLoginStrategy>`로 Bean 등록해 `AuthService`에서 provider로 전략을 O(1) 조회한다.

**SecurityContext**: `JwtAuthenticationFilter`가 `UsernamePasswordAuthenticationFilter` 앞에서 Bearer 토큰을 파싱해 `LoginMember(UUID, MemberRole)`를 `SecurityContext`에 설정한다. 유효하지 않은 토큰은 context를 비우고 다음 필터로 진행(인증 필요 경로에서 401 반환).

**마이그레이션 FK 처리**: `member` 테이블을 참조하는 4개 자식 테이블이 모두 `NO ACTION` FK를 가지고 있어, prod DB에 ANONYMOUS 회원 참조 데이터(submission 123건, member_suspend_history 2건)가 존재했다. 자식 테이블을 순서대로 먼저 삭제한 뒤 member를 삭제하도록 마이그레이션을 구성했다.

## 주의사항

- **Firebase JSON 미발급**: `FIREBASE_SERVICE_ACCOUNT_PROD_JSON` GitHub Secret과 로컬 `firebase-service-account-dev.json`은 Firebase 콘솔에서 직접 발급 후 등록 필요. 발급 전까지 `@Profile("!test")`로 인해 dev/prod 기동 시 오류 발생함
- **APPLICATION_PROD_YML Secret 업데이트 필요**: 기존 Secret에 `jwt` 설정 블록과 `firebase.key-path: classpath:firebase-service-account-prod.json` 추가 필요
- **`LoginMember`, `JwtAuthenticationFilter` 패키지 위치**: 설계 문서에는 `PQL-Common/security`로 계획했으나, `PQL-Common`이 `PQL-Domain-Member`를 의존하지 않는 구조상 `PQL-Domain-Member/auth/presentation/security`로 배치했다
