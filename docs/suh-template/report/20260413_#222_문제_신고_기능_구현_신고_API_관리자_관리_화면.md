# ⚙️[기능추가][문제신고] 문제 신고 기능 구현 (신고 API, 관리자 관리 화면) #222

## 개요

AI가 생성한 문제/선택지에 오류가 있을 때 사용자가 직접 신고할 수 있는 기능을 구현했다. 문제 풀기 결과 화면에 신고 버튼을 추가하고, 복수 선택 가능한 카테고리 기반 신고 모달을 제공한다. 백엔드는 QuestionReport 엔티티와 신고/조회/처리 API를 추가했으며, 관리자는 신고 집계 목록과 상세 화면에서 문제 비활성화 및 Submission 소급 보정(정답 처리)까지 수행할 수 있다.

## 주요 구현 내용

**신고 흐름**: 사용자가 문제 풀기 완료 후 결과 화면에서 신고 버튼 클릭 → `ReportModal`에서 카테고리 복수 선택 → `POST /api/questions/{questionUuid}/report` 호출 → 프론트에서 해당 `submissionUuid`를 완료 목록(Set)에 추가하여 중복 신고 방지. 백엔드는 `UNIQUE INDEX(member_uuid, submission_uuid)`로 DB 수준에서도 중복 차단.

**카테고리 저장**: `@ElementCollection` + `@CollectionTable`로 `question_report_category` 별도 테이블에 정규화. 카테고리 추가/삭제가 스키마 변경 없이 가능.

**관리자 소급 보정**: resolve 요청 시 `CorrectionScope`에 따라 분기.
- `QUESTION_WIDE`: 해당 문제의 모든 오답 제출 → `isCorrect = true` 일괄 업데이트
- `CHOICE_SET_ONLY`: 해당 선택지 세트 기준 오답 제출만 보정
- `NONE`: 상태 변경만 수행

**집계 쿼리**: JPQL이 `SUM(CASE WHEN enum...)` 패턴을 지원하지 않아 네이티브 SQL로 작성. `ReportSummaryRow` inner record에서 `Object[]` 파싱 처리 — Controller에 변환 로직 없음.

**Thymeleaf 규칙 준수**: 신고 집계 화면은 JS fetch + `innerHTML` 렌더링 방식으로 구현. `th:onclick` 없이 `data-uuid` 속성 + JS 이벤트 위임으로 행 클릭 네비게이션 처리.

## 주의사항

- **`categoryDistribution`** `AdminReportSummary` DTO에 포함되어 있으나 현재 API 응답에 항상 빈 맵(`{}`)으로 내려감. 집계 통계가 필요하면 네이티브 쿼리에서 카테고리별 카운트를 추가해야 함.
- **관리자 UUID**: 신고 처리(resolve) 시 `X-Member-UUID` 헤더로 관리자를 식별. 현재 관리자 인증 없이 운영 중이므로 클라이언트 측에서 `localStorage`의 memberUuid를 사용하는데, `memberUuid`가 없으면 UUID 파싱 실패 발생 — 추후 관리자 인증 도입 시 정리 필요.
- **N+1 조회**: 집계 목록 API에서 문제 stem 조회 시 문제 건수만큼 개별 쿼리 발생. 신고 건이 많아지면 배치 조회로 개선 필요.
- **일괄 처리 직렬 호출**: 현재 미처리 신고를 `for...of`로 순차 resolve API 호출 중. 건수 많을 시 bulk resolve 엔드포인트 추가 권장.
