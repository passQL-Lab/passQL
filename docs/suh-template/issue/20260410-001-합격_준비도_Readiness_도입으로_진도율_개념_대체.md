📝현재 문제점
---

- passQL은 AI Agent가 **무제한으로** 문제를 생성하는 서비스라, "전체 N문제 중 M개 완료" 같은 진도율 개념 자체가 성립하지 않는다.
- 그럼에도 현재 홈 화면은 `correctRate`(정답률) 값을 그대로 progress 바의 width로 렌더링하고 있어, 사용자에게 **"전체 중 몇 % 완료"라는 진도율**로 오독된다.
  - 관련: `client/src/pages/Home.tsx` (line 81~84 부근, `correctRate`를 바 width로 사용)
- 무한 문제 환경에서 사용자가 정말 알고 싶은 건 단 하나: **"나 지금 시험 치면 붙을 수 있어?"**
- SQLD/SQLP는 "끝(시험일)이 있는 시험"이라는 특성이 있는데도, 현재 홈 상단 지표가 이 특성을 전혀 반영하지 못한다.
- 백엔드 `GET /api/progress` 응답은 `solvedCount`, `correctRate`, `streakDays` 3개뿐이라 FE가 "합격에 얼마나 가까운지"를 표현할 수 있는 데이터가 없다.
  - 관련: `server/PQL-Domain-Submission/src/main/java/com/passql/submission/service/ProgressService.java`
  - 관련: `server/PQL-Domain-Submission/src/main/java/com/passql/submission/dto/ProgressResponse.java`

🛠️해결 방안 / 제안 기능
---

- "진도율" 개념을 버리고 **합격 준비도(Readiness)** 로 대체한다. 무한 문제 환경에서도 0~100% 게이지가 자연스럽게 성립한다.
- **산식**: `Readiness = Accuracy × Coverage × Recency` (3요소 곱셈, 가중치 없음)
  - **Accuracy**: 최근 50시도의 정답률
  - **Coverage**: 최근 14일 내 1회 이상 푼 "활성 토픽"의 비율 (토픽이 동적으로 추가/수정되므로 가중치 대신 커버리지로 편식 방지)
  - **Recency**: 마지막 학습일 기반 감쇠 (0~1일=1.0 / 2~3일=0.95 / 4~7일=0.85 / 8~14일=0.75 / 15일+=0.70)
- **정직 모드**: 초반 5%, 10% 같은 낮은 수치도 그대로 노출한다. passQL의 "실제 Oracle 실행" 브랜드 철학과 일관됨.
- **투명성**: 최종 점수만이 아니라 3요소 원본값(`accuracy`, `coverage`, `recency`)과 분모/분자(`coveredTopicCount`, `activeTopicCount`, `recentAttemptCount`)를 응답에 포함해, 사용자가 "왜 이 점수인지" 이해할 수 있게 한다.
- **D-day 기반 카피 톤 분기**: 같은 "15%"도 D-60이냐 D-7이냐에 따라 사용자 감정이 다르므로, 백엔드가 `toneKey`를 내려주고 FE가 카피를 매핑한다.
  - `NO_EXAM` / `ONBOARDING` / `POST_EXAM` / `TODAY` / `SPRINT(1~6일)` / `PUSH(7~14일)` / `STEADY(15~30일)` / `EARLY(31일+)`
- **기존 필드 보존**: `solvedCount`, `correctRate`, `streakDays`는 100% 유지하고 `readiness` 블록을 **추가**만 한다. 기존 FE 동작 무영향.
