# ⚙️[기능추가][문제풀이] 문제 풀이 중 이탈 시 경고 모달 구현

## 개요

문제 풀이 도중 다른 화면으로 이동 시도 시 경고 모달을 띄워 풀이 기록 유실을 방지하는 기능을 구현했다. React Router DOM v7의 `useBlocker` 훅으로 앱 내 네비게이션을 가로채고, 공통 `ConfirmModal` 컴포넌트로 이탈 여부를 확인하는 방식이다. 연습 세션(`PracticeSet`), 데일리 챌린지(`DailyChallenge`), 단독 문제 풀이(`QuestionDetail`) 세 화면에 모두 적용했다.

## 변경 사항

### 신규 컴포넌트
- `client/src/components/ConfirmModal.tsx`: 범용 Confirm 다이얼로그 컴포넌트 신규 생성. `isOpen`, `title`, `description`, `confirmLabel`, `cancelLabel`, `onConfirm`, `onCancel` props 지원. 모바일에서는 바텀시트, 데스크톱에서는 중앙 모달 형태로 표시. ESC 키 및 오버레이 클릭으로 닫기 지원.

### 이탈 방지 적용
- `client/src/pages/PracticeSet.tsx`: `useBlocker(!shouldNavigateToResult)` 적용. 마지막 문제 완료(`shouldNavigateToResult === true`) 전까지 이탈 차단. `useEffect`를 훅 규칙에 따라 조건부 return 이전으로 이동.
- `client/src/pages/DailyChallenge.tsx`: `useBlocker(!isLoading && feedback === null)` 적용. 데이터 로딩 중에는 차단하지 않고, 제출 완료(`feedback` 설정) 후 해제.
- `client/src/pages/QuestionDetail.tsx`: `useBlocker(practiceMode !== true && !submitted)` 적용. 단독 풀이 모드(`practiceMode !== true`)에서만 활성화. `submitted` state 추가 — `submitAnswer` 성공 직후 `true`로 설정해 네비게이션 전 차단 해제.

## 주요 구현 내용

**useBlocker 연동 패턴**

React Router DOM v7의 `useBlocker`는 네비게이션 시도 시 `blocker.state`를 `"blocked"`로 전환한다. 이 상태를 `ConfirmModal`의 `isOpen` 조건으로 사용하고, "계속 풀기" 클릭 시 `blocker.reset()`, "나가기" 클릭 시 `blocker.proceed()`를 호출하는 패턴을 세 화면에 공통 적용했다.

**차단 해제 시점 설계**

각 화면의 저장 구조에 맞게 차단 해제 시점을 다르게 설정했다.
- `PracticeSet`: 세션 기록은 마지막 문제 완료 시 결과 페이지로 이동하며 저장되므로 `shouldNavigateToResult`를 해제 기준으로 사용
- `DailyChallenge`: 정답 선택 후 `submitAnswer` API 호출 성공 시 `feedback` state가 설정되므로 이를 해제 기준으로 사용
- `QuestionDetail`: 단독 모드에서 제출 버튼 클릭 후 `submitAnswer` 성공 시 신규 `submitted` state를 `true`로 설정해 해제

**UX 원칙 적용**

파괴적 액션(나가기)을 Secondary 버튼(하단)으로, 유지 액션(계속 풀기)을 Primary 버튼(상단)으로 배치해 실수로 나가는 상황을 방지했다.

## 주의사항

- `useBlocker`는 SPA 내 라우팅만 차단한다. 브라우저 탭 닫기·새로고침(`beforeunload`)은 의도적으로 제외했다 — 과도한 UX 방해로 판단.
- `PracticeSet`에서 `useBlocker`와 `useEffect` 모두 React 훅 규칙상 조건부 return 이전에 위치해야 한다. 초기 구현에서 `useEffect`가 조건부 return 이후에 있어 훅 규칙 위반이 발생했으며, 코드 리뷰에서 발견해 수정했다.
- `DailyChallenge`의 오답 피드백 상태(`feedback.isCorrect === false`)에서 "다시 풀기" 클릭 시 `feedback`이 `null`로 초기화되면서 blocker가 재활성화된다. 이는 의도된 동작이다 — 오답 후 재풀이 중 이탈도 차단해야 하기 때문이다.
