# AI 서버 및 Spring 서버 초기 프로젝트 구성

## 개요

passQL의 두 백엔드 서버(FastAPI 기반 AI 서버, Spring Boot 기반 메인 서버)를 초기화하고, 각 서버에 대한 멀티모듈 구조, Docker 설정, GitHub Actions CI/CD 파이프라인을 구성하였다.

---

## 변경 사항

### AI 서버 (ai/)

- `ai/pyproject.toml`: Python 3.13 / uv 기반 패키지 관리 설정. FastAPI, uvicorn, pydantic-settings, httpx, qdrant-client 의존성 정의
- `ai/src/main.py`: FastAPI 애플리케이션 진입점. lifespan 컨텍스트, 요청 처리 시간 미들웨어, Swagger(`/docs/swagger`) 및 Redoc(`/docs/redoc`) 경로 설정
- `ai/src/apis/ai_router.py`: AI 기능 라우터 (SQL 오류 설명, diff 해설, 유사 문제 검색)
- `ai/src/apis/health_router.py`: 헬스체크 엔드포인트
- `ai/src/core/config.py`, `exceptions.py`, `logging.py`: 환경 설정, 예외 처리, 로깅 초기화 모듈
- `ai/src/models/ai_request.py`, `ai_response.py`: AI 요청/응답 Pydantic 모델
- `ai/src/services/ai_service.py`, `ollama_client.py`, `qdrant_client.py`: Ollama LLM 연동 및 Qdrant 벡터 검색 서비스
- `ai/src/utils/auth.py`: API 키 인증 유틸리티
- `ai/Dockerfile`: Python 3.13-slim 기반, uv 패키지 매니저 사용, 포트 8093 노출

### Spring 서버 (server/)

- `server/settings.gradle`: 멀티모듈 루트 설정. 7개 서브모듈 포함 (`PQL-Common`, `PQL-Domain-*` 4개, `PQL-Application`, `PQL-Web`)
- `server/build.gradle`: 루트 빌드 설정. Spring Boot 3.4.4, Java 21 toolchain, Lombok 전역 적용
- `server/PQL-Common/build.gradle`: 공통 모듈. Web, JPA, Security, Redis, Cache, Actuator, Flyway, Qdrant, Resilience4j 의존성 집약
- `server/PQL-Application/build.gradle`: 유즈케이스 레이어. Domain 모듈들 조합
- `server/PQL-Web/build.gradle`: 웹 레이어 (실행 모듈). `app.jar`로 패키징
- `server/PQL-Web/src/main/resources/application.yml`: 기본 프로파일 설정 (`prod` 활성), Flyway 마이그레이션, Swagger 경로 설정
- `server/PQL-Web/src/main/resources/application-dev.yml`: 로컬 개발용 MariaDB, Redis, Ollama, Qdrant, Resilience4j 설정
- `server/Dockerfile`: Eclipse Temurin 21 JRE Alpine 기반, `app.jar` 실행, 헬스체크 설정 포함

### GitHub Actions 워크플로우

- `.github/workflows/PROJECT-PYTHON-AI-CI.yaml`: `ai/**` 변경 시 Docker 빌드 검증 (push 없음)
- `.github/workflows/PROJECT-SPRING-PASSQL-CI.yaml`: `server/**` 변경 시 Gradle 빌드 검증 (`-x test`, prod 프로파일)
- `.github/workflows/PROJECT-PYTHON-AI-CICD.yaml`: `deploy` 브랜치 기준 AI 서버 시놀로지 NAS 배포
- `.github/workflows/PROJECT-SPRING-SYNOLOGY-PASSQL-CICD.yaml`: `deploy` 브랜치 기준 Spring 서버 시놀로지 NAS 배포

## 주요 구현 내용

**멀티모듈 계층 설계 (Spring)**  
`PQL-Common` → `PQL-Domain-*` → `PQL-Application` → `PQL-Web` 순서로 의존성이 단방향으로 흐르는 클린 아키텍처 기반 멀티모듈 구조를 채택하였다. 공통 인프라 의존성은 `PQL-Common`에 집약하여 중복을 제거하였다.

**AI 서버 서비스 레이어**  
Ollama(LLM)와 Qdrant(벡터 DB)를 각각 독립 클라이언트로 분리하여 AI 서비스 레이어에서 조합하는 구조를 사용하였다. FastAPI의 lifespan 컨텍스트를 활용하여 애플리케이션 초기화/종료 로직을 명시적으로 관리한다.

**CI/CD 분리 전략**  
`main` 브랜치에서는 빌드 검증만 수행하고, `deploy` 브랜치에서만 실제 배포가 트리거되도록 워크플로우를 분리하였다. 환경 변수(`AI_ENV_FILE`, `APPLICATION_PROD_YML`)는 GitHub Secrets에서 주입한다.

## 주의사항

- `server/PQL-Web/src/main/resources/application-dev.yml`에 Ollama API 키가 평문으로 포함되어 있어, `.gitignore` 처리 또는 환경변수 분리가 필요하다.
- Spring CI 워크플로우는 `-x test` 옵션으로 테스트를 건너뛰고 있어, 향후 통합 테스트 환경(MariaDB, Redis 컨테이너) 연동이 필요하다.
- `ai/.env` 파일은 `.dockerignore` 및 `.gitignore` 대상이나, Dockerfile에서 COPY 대상으로 포함되어 있어 이미지 빌드 시 환경 파일이 삽입된다. 프로덕션 환경에서는 Docker secret 또는 환경변수 주입 방식 검토를 권장한다.
