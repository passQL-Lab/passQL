# [버그][문제설계] AI_ONLY 문제 선택지가 SQL 쿼리로 생성되어 문제유형과 불일치

## 개요

`AI_ONLY` + `EXECUTABLE` 유형 문제의 지문은 "다음 SQL의 실행 결과로 올바른 것은?"이었는데, AI가 선택지 body를 실행 결과 텍스트가 아닌 SQL 쿼리 형태로 생성하는 설계 결함이 존재했다. 이를 방치하면 사용자가 각 선택지 실행 버튼을 눌러 결과를 바로 확인할 수 있어 학습 효과가 없어지는 문제가 발생한다. 이에 `AI_ONLY` 정책 자체를 "주어진 SQL과 동일한 실행 결과를 내는 SQL을 고르는" 유형으로 재정의하고, 프롬프트·지문·가이드문서를 일괄 수정했다.

## 변경 사항

## 주요 구현 내용

### 정책 재정의 방향

| 구분 | 변경 전 | 변경 후 |
|------|--------|--------|
| 지문 패턴 | "다음 SQL의 실행 결과로 올바른 것은?" | "다음 SQL과 동일한 실행 결과를 내는 SQL은?" |
| 선택지 body 형태 | (의도치 않게) 실행 결과 텍스트 혼입 | 반드시 실행 가능한 SQL 쿼리 |
| 정답 선택지 | answerSql 실행 결과와 일치하는 1개 | answerSql과 동일한 결과를 내는 다른 SQL 1개 |

### 프롬프트 v3 핵심 변경

시스템 프롬프트에 아래 제약을 명시적으로 추가:

```
각 선택지의 "body" 필드는 반드시 MariaDB에서 실행 가능한 SELECT SQL 쿼리여야 한다.
- 절대 금지: "NAME | DEPT_NAME\n홍길동 | 개발팀" (실행 결과 텍스트)
- 절대 금지: "홍길동, 이철수" (단순 데이터 나열)
```

유저 프롬프트도 "기준 SQL과 동일한 실행 결과를 내는 SQL(정답 1개) + 다른 결과를 내는 SQL(오답 3개) 생성" 방향으로 재작성.

### 기존 데이터 정리

- 기존 오염된 stem(`실행 결과로 올바른 것은?` 패턴)을 `UPDATE question`으로 일괄 수정
- 기존 잘못 생성된 선택지는 `choice_set_item` → `choice_set` 순서로 FK 안전하게 삭제 → 다음 선택지 생성 요청 시 v3 프롬프트로 재생성 유도

## 주의사항

- 이후 `AI_ONLY`가 여전히 "실행 결과 예측" 유형이 필요한 경우에는 별도 `RESULT_MATCH` 정책으로 분리해야 한다 (이슈 #110 참고)
- v3 프롬프트 적용 후에도 AI가 간헐적으로 실행 결과 텍스트를 반환하는 경우에 대비한 Sandbox 검증 강화가 필요하다 (이슈 #103 흐름과 연결)
