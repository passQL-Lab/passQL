# #70 문제 풀기 화면 재설계 — CodeRabbit 리뷰 반영

## 개요

PR #74에 대한 CodeRabbit 코드리뷰 지적 사항을 반영한 수정 작업이다. 제출 경로 SQL 안전성 우회, DTO 타입 불일치, 서비스 트랜잭션 규약 위반, Gemini 예외 미처리, 회원 존재 검증 누락 등 7개 항목을 수정했다. 별개로, 샌드박스 풀 설정값을 app_setting DB에서 실시간으로 읽도록 SandboxPool과 AppInitializer를 개선했다.

---

## 변경 사항

### 1. 보안 / 안전성

- `QuestionExecutionService.java`
  - `submitWithResult()` — `selectedChoiceKey`가 null/blank일 때 NPE 발생하던 문제 수정. null 가드를 먼저 적용하고 `INVALID_INPUT_VALUE` 에러 반환.
  - 존재하지 않는 선택지 키 입력 시 `QUESTION_NOT_FOUND` → `CHOICE_SET_NOT_FOUND`로 에러코드 교정 (의미 불일치 수정).
  - 제출 경로(`selectedSql`, `correctSql`)에 `sqlSafetyValidator.validate()` 추가. 기존에는 `executeChoice()`만 검증하고 제출 경로는 그냥 실행해서 DML/DDL이 sandbox를 오염시킬 수 있었음.

### 2. DTO 타입 규약

- `SubmitResult.java`
  - `boolean isCorrect` → `Boolean isCorrect` (프로젝트 규약: DTO Boolean 필드는 반드시 래퍼 타입 사용).

### 3. 서비스 트랜잭션 규약

- `SubmissionService.java`
  - 클래스 레벨 `@Transactional` → `@Transactional(readOnly = true)` 변경.
  - `recordSubmission()`, `submit()` 변경 메서드에 `@Transactional` override 선언.
  - 읽기 메서드의 중복 `@Transactional(readOnly = true)` 제거 (클래스 레벨 상속으로 커버).

- `AiCommentService.java`
  - 클래스 레벨 `@Transactional(readOnly = true)` 추가.
  - `evictCache()`에 `@Transactional` override 선언.

### 4. 예외 처리

- `AiCommentService.java`
  - `geminiClient.chat()` 호출을 try-catch로 감싸고, 실패 시 `CustomException(ErrorCode.AI_UNAVAILABLE)` 발생. 기존에는 Gemini 장애 시 예외가 그대로 500으로 올라갔음.

### 5. 회원 존재 검증

- `TopicAnalysisService.java`
  - `MemberRepository` 주입 후 `getTopicAnalysis()` 진입 시 회원 존재 여부 검증 (`existsById()`). 존재하지 않으면 `MEMBER_NOT_FOUND` 반환. `ProgressService.getHeatmap()`과 동일한 패턴.

### 6. 쿼리 안정성

- `SubmissionRepository.java`
  - `findTopicStatsAfter()` ROW_NUMBER `ORDER BY submitted_at DESC` → `ORDER BY submitted_at DESC, submission_uuid DESC` tie-breaker 추가. 동일 시각 제출이 존재할 때 집계 결과가 비결정적이던 문제 해결.

### 7. 샌드박스 풀 설정 동적화 (별개 작업)

- `SandboxPool.java`
  - 기존 `@Value`로 yml에서 고정 로딩하던 `concurrency`, `wait_seconds`를 `AppSettingService`를 통해 DB에서 실시간으로 읽도록 변경. 재배포 없이 설정값 조정 가능.

- `AppInitializer.java`
  - 서버 기동 시 `sandbox.pool.concurrency`, `sandbox.pool.wait_seconds` 값을 DB → Redis 워밍업. DB가 비어있으면 yml 기본값(30, 60)을 DB에 적재 후 워밍업. 기존 Gemini API Key 동기화와 동일한 우선순위 패턴 적용.

- `V0_0_45__add_sandbox_pool_settings.sql`
  - `app_setting` 테이블에 `sandbox.pool.concurrency`(30), `sandbox.pool.wait_seconds`(60) 시드 데이터 추가. `ON DUPLICATE KEY UPDATE`로 멱등성 보장.

## 주요 구현 포인트

**SQL 검증 일관성 확보**
`executeChoice()`는 검증하는데 `submitWithResult()`는 그냥 실행하는 비대칭이 보안 구멍이었다. 제출 경로도 동일하게 `sqlSafetyValidator.validate()` → `sandboxExecutor.execute()` 순서를 밟도록 통일했다.

**샌드박스 풀 설정 흐름**
`AppInitializer` → DB/Redis 워밍업 → `SandboxPool.init()` at `@PostConstruct` → `AppSettingService.getInt()`로 Redis에서 읽어 Semaphore 초기화. 설정 변경은 DB를 수정하고 재기동하면 반영된다.

---

## 주의사항

- `SandboxPool`의 `concurrency`는 `@PostConstruct` 시점에 한 번만 읽어 Semaphore를 초기화하므로, 런타임 중 DB 값을 바꿔도 재기동 전까지는 반영되지 않는다. `wait_seconds`는 매 `acquire()` 호출마다 읽으므로 즉시 반영된다.
- `AiCommentService.evictCache()`는 Redis만 삭제하며 JPA 쓰기가 없으므로 `@Transactional`이 실질적 효과는 없지만, 규약 일관성을 위해 선언한다.
