## 개요

설정 화면에 UI는 존재하나 백엔드 미구현 상태였던 건의사항 기능을 완성했다. 사용자 제출/조회 API, 관리자 관리 페이지, Flyway 마이그레이션, 그리고 발생한 프로덕션 버그 수정까지 포함한 전체 구현이다.

## 변경 사항

### Domain — Entity / Enum / DTO

- `server/PQL-Domain-Meta/.../entity/Feedback.java` (신규)

### Domain — Repository / Service

- `server/PQL-Domain-Meta/.../repository/FeedbackRepository.java` (신규)
  - `findByMemberUuidOrderByCreatedAtDesc` — 사용자별 목록 최신순 조회
  - `findAllByOrderByCreatedAtDesc` — 전체 페이지 조회 (관리자)
  - `@Query("SELECT f.status, COUNT(f) FROM Feedback f GROUP BY f.status")` — 상태별 집계 단일 쿼리

- `server/PQL-Domain-Meta/.../service/FeedbackService.java` (신규)
  - `submit()`: content 유효성 검증(빈 값, 500자 초과) 후 저장
  - `getMyFeedbacks()`: 사용자 UUID 기준 목록 반환, 없으면 빈 배열 (404 아님)
  - `getAllFeedbacks()`: 페이지네이션 적용 전체 목록 (관리자)
  - `countByStatus()`: GROUP BY 단일 쿼리로 상태별 카운트를 `Map<FeedbackStatus, Long>` 반환, 데이터 없는 상태는 `0L` 기본값
  - `updateStatus()`: 관리자 상태 변경, 존재하지 않으면 `FEEDBACK_NOT_FOUND` (404)

## 주요 구현 내용

### 사용자 API 흐름

```
POST /api/feedback
  X-Member-UUID: {uuid}
  { "content": "..." }
→ FeedbackService.submit()
  → content 유효성 검증 (빈값 400, 500자 초과 400)
  → Feedback 저장 (status: PENDING)
→ { feedbackUuid, status, createdAt }

GET /api/feedback/me
  X-Member-UUID: {uuid}
→ FeedbackService.getMyFeedbacks()
  → findByMemberUuidOrderByCreatedAtDesc
→ { items: [...] }  // 없으면 { items: [] } (200)
```

### 상태별 카운트 최적화

페이지 로드마다 count 쿼리를 상태별로 3번 날리는 대신 GROUP BY 단일 쿼리로 처리한다.

```java
// FeedbackRepository
@Query("SELECT f.status, COUNT(f) FROM Feedback f GROUP BY f.status")
List<Object[]> countGroupByStatus();

// FeedbackService — 데이터 없는 상태도 0L로 초기화
Map<FeedbackStatus, Long> result = new EnumMap<>(FeedbackStatus.class);
for (FeedbackStatus s : FeedbackStatus.values()) result.put(s, 0L);
feedbackRepository.countGroupByStatus()
    .forEach(row -> result.put((FeedbackStatus) row[0], (Long) row[1]));
```

### 프로덕션 버그: CHAR(36) vs UUID 타입 불일치

`V0_0_115` SQL이 UUID 컬럼을 `CHAR(36)`으로 생성했고, Hibernate 6이 Java `UUID` 파라미터를 PostgreSQL `uuid` 타입으로 바인딩하면서 `operator does not exist: character = uuid` 에러로 500이 반환됐다. `V0_0_119`에서 조건부 타입 변환으로 수정했으며, 상세 분석은 `.report/20260413_#218_feedback_UUID_CHAR36_PostgreSQL_타입불일치_500오류_수정.md` 참고.

## 주의사항

- 이슈 원안의 `FeedbackStatus` 정의는 `PENDING / REVIEWED / APPLIED`였으나, 관리자 화면 구현 시 `RESOLVED`가 혼용되었다. `AdminFeedbackController`의 `statusCounts.get(FeedbackStatus.RESOLVED)` 호출이 있으나 실제 Enum에는 `APPLIED`만 존재한다. 배포 전 Enum 값 통일 필요.
- `GET /api/feedback/me` 에서 UUID 파싱 실패 시 Spring이 400을 반환하나, 에러 응답 형식이 `GlobalExceptionHandler`를 거치지 않아 형식이 다를 수 있다.
