### 📌 작업 개요

홈 화면 API 연동을 위해 단일 `useHome.ts`에 모여 있던 데이터 패칭 로직을 API 도메인별 독립 훅으로 분리. Home 컴포넌트는 각 훅을 병렬 호출하는 조합 구조로 리팩토링되었으며, API 계약 테스트도 추가.

### 🎯 구현 목표

- `useHome.ts` 제거 → 훅 단일 책임 원칙 적용
- 각 훅의 독립적 캐시 관리 및 에러 격리
- 홈 화면에서 필요한 6개 API를 병렬 호출하는 구조 확립
- 프론트-백엔드 API 필드명/타입 계약을 자동 검증하는 테스트 추가

### ✅ 구현 내용

#### 훅 파일 분리 (useHome.ts → 6개 파일)
- **삭제**: `src/hooks/useHome.ts` (59줄 통합 훅)
- **추가**: `useGreeting.ts`, `useTodayQuestion.ts`, `useRecommendations.ts`, `useSelectedSchedule.ts`, `useHeatmap.ts`
- **유지**: `useProgress.ts` (기존 존재)
- 각 훅은 TanStack Query `useQuery` 기반, `staleTime` 5분(일정은 30분), `retry: false`로 통일

#### Home.tsx 리팩토링
- **파일**: `src/pages/Home.tsx`
- **변경 내용**: 각 훅을 개별 import하여 컴포넌트 최상단에서 병렬 호출
- `isLoading` / `isError`는 progress 훅 기준으로 제어, 나머지는 데이터 부재 시 섹션 자체를 숨기거나 로딩 placeholder 표시

#### API 계약 테스트 추가
- **파일**: `src/api/api-contract.test.ts`
- **목적**: 백엔드 스펙과의 field명/타입 불일치를 빌드 타임에 잡기 위한 contract test
- **커버 범위**:
  - `explainError` — body 필드가 camelCase(`errorMessage`)인지 검증
  - `diffExplain` — body 필드(`questionUuid`, `selectedChoiceKey`)가 snake_case 아님을 검증
  - `submitAnswer` — body에 `selectedChoiceKey`만 포함, 파라미터 2개 확인
  - `fetchHeatmap` — query param에 `memberUuid/from/to` 포함 여부
  - 타입 export 정합성 (SSE 관련 타입 제거 확인 등)

#### types/api.ts 정리
- `HeatmapEntry`, `HeatmapResponse` 타입 추가 (#42 연동용)
- 불필요한 SSE 관련 타입(`SubmitPayload`, `ChoiceGenerationPhase` 등) 제거

### 🔧 훅 구조 상세

| 훅 | API | queryKey | staleTime | 인증 |
|---|---|---|---|---|
| `useGreeting` | GET /home/greeting | `["greeting", uuid]` | 5분 | uuid query param |
| `useTodayQuestion` | GET /questions/today | `["todayQuestion", uuid]` | 5분 | uuid query param |
| `useRecommendations` | GET /questions/recommendations | `["recommendations"]` | 5분 | - |
| `useSelectedSchedule` | GET /exam-schedules/selected | `["selectedSchedule"]` | 30분 | - |
| `useHeatmap` | GET /progress/heatmap | `["heatmap", uuid, from, to]` | 5분 | uuid query param |
| `useProgress` | GET /progress | `["progress"]` | - | uuid query param |

Home 진입 시 6개 훅이 동시에 발화하여 병렬 패칭.
