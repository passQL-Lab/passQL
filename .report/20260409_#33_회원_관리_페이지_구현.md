## 2. 핵심 동작 요약

### 2-1. 제재(Suspend)
1. 관리자가 상세 페이지 모달에서 사유 + 만료일시 입력
2. `POST /admin/members/{uuid}/suspend`
3. 서비스 레이어에서 `suspendUntil` null / 과거 시각 검증 → `INVALID_SUSPEND_UNTIL(400)` 거부
4. `member.status = SUSPENDED`, `member.suspendUntil` 설정, 이력 저장

### 2-2. 해제(Unsuspend) — 3가지 경로
| 경로 | 트리거 | 처리 |
|---|---|---|
| 수동 | `POST /admin/members/{uuid}/unsuspend` | 즉시 해제 |
| 자동 배치 | `@Scheduled(fixedDelay = 60_000)` | `suspend_until < now()` AND `status = SUSPENDED` 배치 조회 후 건별 해제 |
| 접속 시 | `getMe()` 호출 시 | 만료됐으면 즉시 해제, 만료 전이면 **`403 MEMBER_SUSPENDED` throw** |

### 2-3. 검색
- JPA Specification 기반 동적 쿼리 (QueryDSL 미도입)
- 필터: 닉네임 부분일치, 상태, 역할, 가입일 범위, 마지막 접속일 범위, 테스트 계정 포함 여부
- 페이지네이션 20건/페이지, 최대 100건

## 3. 정적 리뷰 결과 & 자체 반영

### 🚨 Critical

| # | 이슈 | 처리 |
|---|---|---|
| **C1** | 어드민 엔드포인트 인증 부재 (`SecurityConfig.anyRequest().permitAll()` + CSRF disabled) | **의도적 스킵** — 프로젝트 전체가 mock 인증 전제. 권한 체계 도입 시 일괄 처리 예정 |
| **C2** | `getMe`에서 **만료 전 SUSPENDED 회원도 정상 통과** → 제재 무력화 | ✅ **차단 로직 추가** — `ErrorCode.MEMBER_SUSPENDED(403)` 추가, 만료 지난 경우만 자동 해제, 그 외는 예외 throw |

### ⚠️ Major (전부 해결)

| # | 이슈 | 처리 |
|---|---|---|
| **M1** | Thymeleaf `th:each + th:if` 동일 엘리먼트 앤티패턴 (list/detail 3곳) | ✅ `th:if` 제거 — 빈 컬렉션이면 `th:each` 자체가 돌지 않음 |
| **M3** | 페이지네이션 링크가 `nickname/status/role`만 전달 → 날짜·테스트계정 필터 유실 | ✅ `joinedFrom/To`, `lastSeenFrom/To`, `includeTest` 모두 링크에 포함 |
| **M4** | `suspendUntil` null / 과거 시각 검증 없음 → 실수로 영구제재 가능 | ✅ 서비스 레이어에서 `null` + `isBefore(now())` 거부 (`INVALID_SUSPEND_UNTIL`). jakarta-validation 프로젝트 전체 미도입이므로 수동 검증 선택 |
| **M5** | `autoUnsuspendExpired` 배치 — 한 건 실패가 전체 롤백 | ✅ `REQUIRES_NEW` + **self-injection(`@Lazy`)** + UUID 재조회로 detached race 방지. 건별 try-catch로 실패 격리, `succeeded/failed` 로깅 |

### 💡 Minor (일부 해결, 나머지 후속 이슈)

| # | 이슈 | 처리 |
|---|---|---|
| **m2** | `page` 음수 허용 → `PageRequest.of`에서 `IllegalArgumentException` | ✅ `Math.max(0, page)` + size 하한 `Math.max(1, size)` |
| **m8** | 테스트 내부 메서드에 중첩 `@Transactional` (self-invocation이라 무의미) | ✅ 제거 |
| m5 | `MemberSuspendHistory.actedAt` vs `BaseEntity.createdAt` 중복 | 🔄 후속 이슈 — `BaseEntity` 제거 또는 `actedAt` 제거 택일 |
| m6 | 이메일·마지막 IP 등 PII 노출 | 🔄 C1 (인증 가드)과 묶여야 의미 있음 |
| m7 | `badge-warning` 암묵 분기 — 향후 `MemberStatus` 값 추가 시 모두 warning | 🔄 현 enum 4개 고정이라 당장 문제 없음 |
| m9 | 테스트가 로컬 DB 데이터 의존 — 회원이 없으면 SKIP 구조 | 🔄 테스트 픽스처 도입 권장 (별도 이슈) |

## 4. 적용 시 주의사항

### 4-1. 프론트엔드 작업 필요 — `MEMBER_SUSPENDED(403)` 핸들링
이제 제재 중 회원이 `/api/members/me`를 호출하면 **403**이 나갑니다.
- 프론트에서 이 에러 코드를 "제재됨" 화면으로 라우팅 필요
- 만료 시점이 지나면 같은 호출이 자동 해제 후 200을 반환하므로 재시도 가능

### 4-2. M5 self-injection 구조
`@Transactional(REQUIRES_NEW)`가 Spring AOP 프록시를 거치도록 self-injection 패턴 사용:

```java
@Autowired
@Lazy
private MemberAdminService self;

public void autoUnsuspendExpired() {
    // ...
    for (UUID uuid : expiredUuids) {
        try {
            self.unsuspendSingleInNewTx(uuid);  // 반드시 self. 경유
            // ...
        } catch (Exception e) { /* 격리된 실패 로깅 */ }
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void unsuspendSingleInNewTx(UUID memberUuid) { ... }
```

**주의:** 스케줄러가 호출하는 `autoUnsuspendExpired()` 내부에서 단건 메서드를 호출할 때 **반드시 `self.`를 거쳐야** `REQUIRES_NEW`가 동작합니다. `@RequiredArgsConstructor`는 `final` 필드만 대상이라 `self`는 생성자 밖에서 주입되므로 순환 참조 없음.

### 4-3. validation 프레임워크 미도입
`jakarta.validation` 의존성이 프로젝트 전체에 없어 `@Valid/@NotNull` 대신 서비스 레이어 수동 검증. 전 어드민 공통 input 검증 체계를 잡는다면 별도 이슈로 트래킹 권장.

## 5. 잘한 점

- **모듈 경계 준수** — 도메인 로직은 `PQL-Domain-Member`, 컨트롤러·스케줄러는 `PQL-Web`
- **정적 팩토리** — `MemberSuspendHistory.ofSuspend/ofUnsuspend`로 유스케이스별 생성 의도 명확
- **DTO는 record로 통일** — 불변성 + 보일러플레이트 감소
- **`@Transactional(readOnly=true)` 클래스 레벨 + 쓰기 오버라이드** — 기존 `MemberService` 스타일과 일치
- **JPA Specification null-safe 조립** — QueryDSL 없이도 깔끔한 동적 검색
- **Flyway 버전 네이밍 일관성** (`V0_0_24`)
- **`@EnableScheduling` 누락 방지**
