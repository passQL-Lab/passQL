# #68 문제 풀기 카테고리 선택 화면 Topic API 연동

### 📌 작업 개요

`/questions` 진입 시 BE `/api/meta/topics` API를 사용하지 않고 FE 임의 Mock 코드(JOIN, SUBQUERY, GROUP_BY 등)로 카테고리를 표시하던 문제 해결. FE Mock 코드 체계를 BE DB 시드 9개 topic 코드 체계로 동기화하고, 카테고리 그리드 화면을 새로 구현하여 BE API 연동 완료.

---

### 🔍 문제 분석

**FE vs BE 코드 불일치 (수정 전)**

| FE Mock code | BE DB code | 비고 |
|---|---|---|
| JOIN | sql_join | code 불일치 |
| SUBQUERY | sql_subquery | code 불일치 |
| GROUP_BY | sql_group_aggregate | code 불일치 |
| DDL | sql_ddl_dml_tcl | code 불일치 |
| CONSTRAINT | (BE에 없음) | BE에는 독립 topic 아님 |
| (FE 누락) | data_modeling | FE 미반영 |
| (FE 누락) | sql_basic_select | FE 미반영 |
| (FE 누락) | sql_function | FE 미반영 |
| (FE 누락) | sql_window | FE 미반영 |
| (FE 누락) | sql_hierarchy_pivot | FE 미반영 |

- FE 5개 임의 코드 → BE 공식 9개 코드 체계로 전면 교체 필요
- `/questions` 진입 시 카테고리 그리드 화면 없이 바로 문제 목록으로 이동하는 구조였음

---

### ✅ 구현 내용

#### 1. MOCK_TOPICS — BE 9개 topic 코드 체계로 교체

- **파일**: `src/api/mock-data.ts`
- **변경 내용**: 12개 FE 임의 코드 배열 → BE DB 시드 9개 topic으로 전면 교체
- **BE topic 목록**: `data_modeling`, `sql_basic_select`, `sql_ddl_dml_tcl`, `sql_function`, `sql_join`, `sql_subquery`, `sql_group_aggregate`, `sql_window`, `sql_hierarchy_pivot`
- 각 topic에 `topicUuid`, `code`, `displayName`, `sortOrder`, `isActive`, `subtopics` 구조 그대로 유지

#### 2. MOCK_QUESTIONS — topicCode/topicName 동기화 및 필터링 로직 수정

- **파일**: `src/api/mock-data.ts`
- **변경 내용**:
  - `topicCode`: `JOIN` → `sql_join`, `SUBQUERY` → `sql_subquery`, `GROUP_BY` → `sql_group_aggregate`, `DDL` → `sql_ddl_dml_tcl`, `CONSTRAINT` → `sql_basic_select`
  - `topicName`: FE 임의 이름 → BE `displayName` 값으로 동기화 (e.g., `"JOIN"` → `"조인 (JOIN)"`)
  - 필터링 로직 개선: `topicDisplayName` 중간 변환 제거 → `q.topicCode === topic` 직접 비교로 단순화

```typescript
// 수정 전 (brittle): displayName 중간 변환
const topicDisplayName = topic ? MOCK_TOPICS.find((t) => t.code === topic)?.displayName : undefined;
const filtered = topicDisplayName ? MOCK_QUESTIONS.filter((q) => q.topicName === topicDisplayName) : topic ? [] : MOCK_QUESTIONS;

// 수정 후: topicCode 직접 비교
const filtered = topic ? MOCK_QUESTIONS.filter((q) => q.topicCode === topic) : MOCK_QUESTIONS;
```

#### 3. Questions.tsx — 카테고리 그리드 화면 구현 및 API 연동

- **파일**: `src/pages/Questions.tsx`
- **변경 내용**: 기존 단일 뷰(필터 + 목록)에서 URL search param 기반 두 뷰 전환 구조로 리팩토링

**카테고리 그리드 뷰** (`/questions` — topic 파라미터 없을 때):
- `useTopics()`로 BE API 호출, `isActive` 필터 + `sortOrder` 정렬
- 2열(모바일) / 3열(태블릿 이상) 반응형 그리드
- 각 카드: topic `displayName` + 서브토픽 개수 표시
- 로딩 시 skeleton, 에러 시 `ErrorFallback` (refetch 방식)

**문제 목록 뷰** (`/questions?topic=sql_join` — topic 파라미터 있을 때):
- "토픽 전체" 뒤로가기 버튼 (ArrowLeft 아이콘)으로 그리드로 복귀
- topic 헤딩 + 난이도 필터만 표시 (topic 드롭다운 제거)
- 빈 목록 empty state 추가

**URL 기반 상태 관리**:
- `useState(topic)` → `useSearchParams()` 교체로 URL deep-link 지원
- 브라우저 뒤로/앞으로 이동 시 그리드 ↔ 목록 전환 정상 동작

#### 4. useQuestions — enabled 옵션 추가

- **파일**: `src/hooks/useQuestions.ts`
- **변경 내용**: `enabled?: boolean` 파라미터 추가, `useQuery`에 전달
- **이유**: topic 미선택 상태(카테고리 그리드 뷰)에서 `GET /questions` 불필요한 API 호출 방지
- 기존 호출부 하위 호환성 유지 (`enabled` 미지정 시 기본값 `true`)

### 🔧 주요 변경사항 상세

#### Questions.tsx 뷰 전환 구조

```
/questions (topic 없음)
  └─ 카테고리 그리드 뷰
       └─ 카드 클릭 → setSearchParams({ topic: code })

/questions?topic=sql_join
  └─ 문제 목록 뷰
       └─ "토픽 전체" 클릭 → setSearchParams({})
```

`topic` 값은 URL 쿼리 파라미터로만 관리되므로 새로고침 시에도 상태 유지, 특정 토픽 URL 직접 공유 가능.

**특이사항**:
- `toSorted()`를 사용하여 `filter()` 결과 배열을 불변하게 정렬 (`.sort()` 대신)
- `refetchTopics()`를 `void`로 래핑하여 onClick 타입 불일치 방지

### 📌 참고사항

- `src/types/api.ts`의 `TopicTree`, `SubtopicItem` 타입은 이미 BE 응답 필드(`topicUuid`, `sortOrder`, `isActive`)를 포함하고 있어 타입 변경 불필요
- `src/api/meta.ts`의 `fetchTopics()`, `src/hooks/useTopics.ts`도 이미 구현되어 있어 그대로 활용
- BE 연동 시 `GET /api/meta/topics` 응답의 `isActive: false`인 topic은 그리드에서 자동 제외됨
