# 문제 풀기 화면 재설계 — 실행 결과 비교 및 자유 SQL 실행 지원

## 개요

문제 풀기 화면의 백엔드를 재설계했다. 기존에는 선택지 클릭 시 `questionUuid`를 샌드박스 DB 이름으로 잘못 전달하는 버그가 있었고, 제출 후 응답에 실행 결과가 포함되지 않아 프론트에서 "내 SQL vs 정답 SQL" 비교가 불가능했다. `QuestionExecutionService`를 실제 구현하고, `SubmitResult`를 확장하여 `EXECUTABLE` 문제에서 선택지 실행 결과와 정답 실행 결과를 함께 반환하도록 개선했다. `CONCEPT_ONLY` 문제는 SQL 실행 없이 키 비교만 수행한다.

## 주요 구현 내용

### 버그 수정: execute 엔드포인트의 dbName 오용
기존 코드는 `sandboxExecutor.execute(questionUuid.toString(), sql)`로 UUID 문자열을 MariaDB 데이터베이스 이름으로 전달했다. `SandboxPool`은 `sandbox_XXXXXXXX` 형태의 이름을 기대하므로 실제 데이터베이스를 찾지 못했다. `question.getSandboxDbName()`을 사용하도록 수정했다.

### EXECUTABLE / CONCEPT_ONLY 분기
`submitWithResult()`에서 `question.getExecutionMode()`로 분기한다.
- `EXECUTABLE`: 선택지 `body`(SQL)와 `question.answerSql`을 각각 샌드박스에서 실행 → `selectedResult`, `correctResult` 포함 반환. `sandboxDbName`이 없거나 SQL이 blank인 경우 해당 결과만 null로 반환 (graceful).
- `CONCEPT_ONLY`: SQL 실행 없이 `isCorrect`, `correctKey`, `rationale`만 반환.

### 자유 SQL 실행 재사용
`executeChoice(questionUuid, sql)`는 선택지 자동 실행뿐 아니라 프론트의 SQL Playground에서도 그대로 사용 가능하다. `CONCEPT_ONLY` 문제에서 호출하면 400 `INVALID_EXECUTION_MODE`를 반환하여 잘못된 호출을 차단한다.

## 주의사항

- `correctResult`는 정답/오답 여부와 무관하게 항상 실행된다. 프론트 UX에서 정답 시에도 정답 SQL 결과를 보여주는 구조라면 현재가 맞다. 오답 시에만 필요하다면 `!isCorrect` 조건 추가로 최적화 가능.
- `ExecutionLog` 저장 로직은 이번 구현에서 제외됐다. `/api/questions/{uuid}/execute` 호출 시 실행 로그가 쌓이지 않으므로 `/admin/monitor`의 통계가 이 경로를 집계하지 못한다. 별도 이슈로 처리 필요.
- `schema_sample_data` 컬럼 추가(V0_0_44)는 Flyway 마이그레이션으로 반영된다. 기존 문제 행은 NULL로 채워지며, 관리자 페이지에서 직접 입력 후 사용한다.
